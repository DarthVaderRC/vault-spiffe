01 · Problem

Identity is only valuable when another system can turn it into a safe access decision.

This repo is opinionated about the gap customers actually feel. A portable workload identity is useful, but by itself it does not authorize a payment API to access backend, or broker short-lived database access. Vault becomes the control plane when it can accept a trusted workload input and turn that input into an outcome outside Vault.

Operating Model: Vault consumes trusted workload identity, issues the right artifact for the next hop, and validates it when another system consumes that artifact.
Identity

Identity proves who the workload is.

In this repo, that trusted input comes from Kubernetes service account tokens on the default path or from SPIRE-issued JWT-SVIDs on the overlay path.

Authorization

Authorization decides what that workload can do.

That is where Vault acts as an identity broker 0 not as a generic token vending machine, but as the control plane that maps trusted identity to policy-backed outcomes.

Proof

The proof point has to live outside Vault.

The demo is only credible when the result is a payment-status API, a relationship-insights API, or a database login that another system actually consumes.

02 · Foundations

Use Kubernetes as the starting trust source, then choose the identity artifact that fits the next protocol boundary.

The first design choice in this demo is not “X.509 or JWT?” It is “what do we already trust?” The default answer is Kubernetes. From there, Vault issues either an X.509 identity for mTLS or a JWT-SVID for an API verifier.

Repository baseline: the runnable lab uses Vault Enterprise because the SPIFFE auth and SPIFFE secrets engine are Enterprise only features.
Default path

Vault Enterprise + Kubernetes

The demo deployes a Vault Cluster in Docker Compose, adds a local kind cluster, and configures auth/kubernetes against a real Kubernetes API.

Artifact fit

Protocol fit beats one-artifact purity.

Use PKI when the next hop is mTLS. Use the SPIFFE secrets engine when the next hop is a relying party that already understands discovery and JWKS.

Boundary

SPIRE enters when workload attestation matters.

The overlay shows where Vault and SPIRE fit together cleanly. It does not pretend Vault replaces SPIRE’s workload attestation model.

Canonical issuer

vault.demo.internal stays stable.

The lab keeps https://vault.demo.internal:18200 as the canonical issuer for TLS and OIDC metadata, while pods reach the host-side Vault cluster through host.docker.internal.

Trust domains

The repo uses two trust domains on purpose.

hashibank.demo belongs to the Vault-native path. spire.hashibank.demo belongs to the SPIRE overlay. That keeps the trust bundles honest.

03 · Zero-trust mTLS

Scenario 1 is not “one client cert.” It is two microservices proving each other’s SPIFFE IDs before a payment response is returned.

The scenario gets more realistic when both sides authenticate to Vault with Kubernetes auth, both receive short-lived X.509 identities, and both participate in the trust decision. That is the difference between a certificate demo and a zero-trust service boundary.

The business proof is a settlement-status API response. The important step is not just certificate issuance; it is explicit peer identity verification during the handshake.
What happens

Both workloads start from Kubernetes identity.

payments-api and mtls-backend each log in through auth/kubernetes, then request exact SPIFFE URI SANs through pki/.

What to notice

Zero trust is enforced on both sides.

The client does not settle for TLS reachability. It verifies the backend SPIFFE ID. The backend does not settle for “any cert from the CA.” It authorizes the caller SPIFFE ID.

Run it with ./scripts/demo-k8s-mtls.sh. The final checkpoint proves the point by returning payment status only after mutual identity validation succeeds.
04 · Cross-network JWT

Scenario 2 turns JWT-SVID validation into a business decision, not a token echo.

The more realistic question is not whether a relying party can verify the signature. It is whether the relying party can use workload identity plus business metadata to decide what response is safe to return.

The audience is relationship-insights-api. The proof point is masked relationship insights and a next-best action, not a claims echo.
What the API enforces

Signature validation is the floor, not the finish line.

The consumer validates the JWT through discovery and JWKS, then authorizes the subject and business claims: application, line_of_business, and customer_data_domain.

Business proof

The response looks like a banker workflow.

The API returns masked relationship insights plus a next-best action, and the assistant UI renders that result at http://localhost:18082/.

Run it with ./scripts/demo-k8s-jwt.sh. The scenario shows the entire verification chain live: minting, discovery, JWKS retrieval, claim authorization, and the assistant-ready response.
05 · SPIRE to Vault

The SPIRE overlay proves that Vault can consume attested workload identity without forcing a second, unrelated auth contract.

This is the clean SPIRE-to-Vault story in the repo today: a SPIRE-issued JWT-SVID is accepted by Vault’s SPIFFE auth method, and Vault turns that trusted identity into short-lived database credentials for a concrete fraud-data query.

The outcome is not “Vault accepted a JWT.” The outcome is “Vault accepted a SPIRE-issued workload identity and turned it into short-lived database access.”
06 · Upstream Authority

The upstream-authority story is about CA chaining, not JWT publication.

The repo’s fourth scenario keeps the boundary crisp: Vault remains the enterprise X.509 trust anchor while SPIRE continues to issue workload certificates. That is a complementary architecture, not a replacement story.

What the repo proves here is trust-chain validation back to the Vault-managed root. It does not claim JWT signing-key publication through the upstream-authority plugin.
07 · AWS Extension

The AWS federation pattern is credible only when the issuer is public and AWS can trust it.

The follow-on design is straightforward conceptually: mint a Vault JWT-SVID, exchange it through AWS web identity federation, and use the short-lived AWS credentials against an S3 bucket. The hard part is not the STS call. The hard part is the issuer shape and IAM trust configuration.

This repo intentionally keeps the pattern deferred until the OIDC issuer is publicly reachable and the IAM trust boundary is real.
What is required from AWS

You need five concrete building blocks.

  • a public HTTPS issuer URL that exposes Vault’s discovery and JWKS endpoints
  • an AWS IAM OIDC provider pointing at that issuer
  • an IAM role trust policy constrained to the expected audience and SPIFFE subject
  • an S3 bucket plus a minimal policy for list or read access
  • AWS CLI or SDK access from the calling workload so the demo can perform the STS exchange and the S3 operation
What still needs a design choice

The issuer path is the real decision.

AWS cannot trust vault.demo.internal or any self-signed local endpoint. The clean implementation needs a public domain, a trusted certificate chain, and a stable route to the OIDC metadata.

Practical answer: if you want this next, the repo needs a public issuer shape first, then an IAM OIDC provider, then the STS role and bucket policy. The STS call itself is the easy part.

The repo is intentionally opinionated about operator flow: bootstrap the environment, review the state, then drive the scenarios through the checkpointed scripts so the proof stays reproducible.
